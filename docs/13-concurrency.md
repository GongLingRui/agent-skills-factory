# 13. 并发与降级策略

> 版本：v0.6 · 2026-05-06

---

## 资源拆池

| 资源 | 控制方式 |
|------|---------|
| LLM | 队列、优先级、token 预算、fallback |
| Embedding | 批处理、缓存、低优先级队列 |
| Rerank | 可降级、可跳过 |
| 知识检索 | 连接池、数据域过滤 |
| 文档解析 | 独立 worker 池 |
| 脚本 worker | 受控池、超时、无网络 |
| 内部 API | 熔断、重试、限速 |

---

## 并发等级

| 等级 | 延迟要求 | 典型场景 |
|------|---------|---------|
| **interactive** | 低延迟（<3s） | 普通问答 |
| **document** | 中等延迟（<30s） | 文档解析和报告 |
| **batch** | 可排队（<5min） | 批量材料处理 |
| **privileged** | 最高优先级 | 关键业务 |

---

## 高峰期降级（按顺序执行）

1. 降低 rerank 数量或跳过 rerank
2. 降低检索 top_k
3. 切到小模型
4. 限制 max_turns
5. 关闭非必要工具
6. 长文档任务转异步

---

## 降级触发条件与自动恢复

### 触发指标（任一超阈值则启动对应级降级）

| 触发条件 | 触发降级动作 |
|----------|-------------|
| LLM 队列 P99 延迟 > 30s | 跳过 rerank |
| LLM 队列 P99 延迟 > 60s | 检索 top_k 从 20 降到 5 |
| LLM 队列 P99 延迟 > 120s | 切到小模型 |
| 错误率 > 5% | 限制 max_turns 到 3 |
| 内部 API 熔断打开 | 关闭非必要工具 |
| 文档解析队列 > 100 任务 | 长文档转异步 |
| CPU 使用率 > 85%（持续 2 分钟） | 跳过 rerank + 限制 max_turns 到 3 |
| 内存使用率 > 90%（持续 2 分钟） | 关闭非必要工具 + 长文档转异步 |

### 自动恢复

- 触发指标连续 5 分钟低于阈值 → 解除最严的一级
- 全部指标正常 + 用户对话失败率 < 1% → 完全恢复

### 控制面

- 运维有手动开关（强制某级别 / 强制恢复 / 屏蔽某 Agent）
- 降级状态对所有 Agent 生效，但 concurrency.queue_priority 高的 Agent 优先获得资源
- 降级状态对前端可见——widget 顶栏显示"系统繁忙，已启用快速模式"提示

### Agent 级降级豁免

部分关键 Agent（如领导审批助手、应急指挥助手）在全局降级时不应被降级策略影响。

**豁免机制**：

- `agent_apps` 表新增字段 `degradation_exempt: boolean`，默认 `false`
- 当全局降级触发时，Agent Runner 先检查当前 Agent 的 `degradation_exempt`：
  - `true` → 该 Agent 的请求**不应用**模型降级（仍用 default 模型）、不关闭工具、不限制 max_turns
  - `false` → 正常执行降级策略
- 豁免范围：仅豁免"有损降级"（切小模型、关工具、限轮数），**不豁免安全降级**（如错误率过高时的熔断保护仍生效）
- 权限：`degradation_exempt` 字段仅 `platform_admin` 可修改，变更写入 `config_change_logs`

**建议豁免列表**：

| Agent 类型 | 建议豁免 | 理由 |
|-----------|---------|------|
| 领导审批类 | 是 | 不可因系统繁忙降低回答质量 |
| 应急指挥类 | 是 | 时效性优先，必须保持完整能力 |
| 普通问答类 | 否 | 降级体验可接受 |
| 批量处理类 | 否 | 异步执行，可排队 |

## 为什么不需要每用户一个容器

- Agent 配置是静态文件，可缓存
- Skill 是声明式文本和 manifest
- Tool 执行由统一网关控制
- 脚本只通过受控 worker 跑
- 用户隔离靠权限 / 数据域 / RunSpec / 审计，不是靠容器

更适合资源受限的内网部署。

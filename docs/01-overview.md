# 01. 项目概述与核心思想

> 版本：v0.6 · 2026-05-06

---

## 一句话讲清楚

我们要做一个 **Agent 应用工厂**，给央企内部各部门快速生产业务 Agent（合同审查、制度问答、会议纪要、材料起草、舆情简报、合规检查等）。

**工厂的核心规则只有一条：**

> **每个 Agent = 一份配置文件 + 一个 Skill 包。Skill 不在运行时被动态选择，而是写死在 Agent 配置里。**

这条规则把所有不确定性赶到边界外，让我们能**像生产订单一样批量生产 Agent**，又能**逐个审计、逐个限流、逐个升级**。

---

## 1. 背景与目标

**用户**：央企内部员工、业务部门、管理部门。

**主要场景**：

- 内部制度、流程、规范、历史材料的检索问答
- 合同、公文、会议纪要、汇报材料、舆情简报的生成与改写
- 基于内部知识库、文档解析、规则工具的轻量业务处理
- 不同部门快速生成专用 Agent

**关键约束（这些约束决定了为什么要轻量）**：

- **服务器资源有限**——做不了"一人一个容器 / 沙箱"的奢侈方案
- **多用户、多部门、多并发**——必须共享资源
- **私有化部署**——内网、本地模型、国产模型或 OpenAI 兼容网关
- **权限、审计、数据域隔离、工具调用管控**（央企硬合规）
- 需要 tool use 和轻量编排，但不需要 Claude Code 那种代码运行时
- **业务 Agent 数量要爆炸式增长，同时月活用户也要起来**

---

## 2. 不做的事（明确边界）

下面这些事**这个项目不碰**：

- 不做 Claude Code 那种个人开发者 CLI
- 不做"一人一容器 / 一人一文件系统"的重运行
- **不做以 RAG 为中心的重平台**——RAG 是独立项目，本系统通过 Tool Gateway 暴露 `kb.search` 接入
- **不做 Skill 创建工具**——Skill 由独立 Skill Creator 系统产出，本系统只消费
- **不做 multi-skill agent**——P0/P1 硬约束：一个 Agent 严格挂载一个 Skill
- 不做任意代码插件市场
- 不做动态 Skill 自动匹配（用户选 Agent 即选 Skill）
- 不做多 Agent 自主协作 / 开放式任务规划

**不是能力不足，是这些都会在企业内网把工程复杂度、安全边界和并发成本拉爆。**

---

## 3. 三个核心想法

### 3.1 一个 Agent = 一份岗位说明书（不是一个员工）

通俗讲：**Agent 不是一个常驻员工，是一份岗位说明书 + 一份操作手册**。

> Agent App = agent.yaml + 一个 Skill + 工具策略 + 知识范围 + 输出格式 + 运行限制

请求来了，系统照说明书 → 现场组装一份"出厂订单"（RunSpec）→ 共享底座按订单干活。

**为什么这样**：企业内网资源有限，不能给每个 Agent 配一个常驻进程或容器。

**P0/P1 硬约束**：一个 Agent 绑定**一个业务 Skill Package**。Skill Package 是**插件化能力包**，内部通过 progressive disclosure 管理 references / scripts / schemas / examples / evals——**不是单文件 prompt**。

**禁止**：

- 运行时动态 Skill 匹配
- 模型从全局 Skill 池自由挑选 Skill

**未来**（P3 之后）：当多个 Skill Package 之间出现高频重复内容时，引入 shared references / shared scripts / capability library——但 **P0/P1/P2 阶段坚决不做**。multi-skill / 子路由 / 动态选择由当时的实战数据决定要不要做。

### 3.2 Skill Package 是插件化能力包（不是单文件 prompt）

Skill Package 是一个**目录**，里面有主文档 + 配套资源 + 可选脚本 + 评测用例。**它不是**一个 prompt 字符串、也不是一个插件运行时。**Skill Package 不能自己启动进程、装依赖、跑代码**。Skill 只描述：

> 我能做什么 / 什么时候用我 / 我要哪些工具 / 我要查哪些知识 / 输出长啥样。

请求开始时，**Skill Compiler（装配车间）** 把 Skill 编译成：

- prompt 片段
- 允许的工具清单
- 允许查的知识范围
- 输出格式
- 运行限制
- 可选的预处理 / 后处理声明

**真正干活的是统一的 Agent Runner 和 Tool Gateway。Skill 自己不直接执行任何东西。**

### 3.3 多并发靠共享底座（不靠多容器）

不是"启动多个 Agent 进程"，是把不同的资源拆成共享池：

- API 层无状态
- Agent 配置常驻缓存
- LLM 调用进队列
- Tool 调用进网关
- 文档解析、检索、rerank、脚本各有自己的资源池
- 按部门、用户、Agent、任务类型限流和分优先级

类比：**一个公司不用给每个员工分一间独立办公室，是给他们配共享会议室、共享打印机、共享网络。哪怕同时来 100 个人，也是同一栋楼撑起来。**

---

## 4. 一句话定义

> **一个面向企业内网的轻量级 Agent 应用工厂。**
>
> 用声明式 Agent Manifest 和 Skill Compiler，把业务能力编译成可审计、可限流、可权限控制的 RunSpec，由共享 Agent Runner、Tool Gateway 和 Model Queue 执行。
>
> **Skill 不在运行时被动态匹配，而是写死在 Agent 配置里**——这条规则是工厂能批量化生产 Agent 的根本前提。

**核心价值**：

- **配置即 Agent**——快速生产大量业务 Agent
- **共享底座**——支撑多并发不靠多容器
- **Tool Gateway 硬校验**——保证工具调用可控
- **Skill Compiler**——避免重插件运行时
- **references/**——承载长知识和范例
- **受控 scripts/**——承担必要的预处理 / 后处理 / 评测
- **RunSpec**——让每次执行可追踪、可审计、可复现
- **终端分层存储**（轻偏好 localStorage + 对话 IndexedDB + 敏感附件不落盘）——业务对话主体不上服务器，合规边界清晰
- **retention gate**——保证数量 KPI 不被噪音稀释、MAU KPI 不被低质量 Agent 拖垮
